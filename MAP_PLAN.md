## 2. פתרון הסתירה: Walk-in מול "לחיצה לא עושה כלום"

**הבעיה:** אמרת שלחיצה על שולחן לא תעשה כלום, אבל גם שהמארחת צריכה לסמן walk-in על שולחן פנוי.

**הפתרון המוצע:**

יש בפינת המסך כפתור "➕ Walk-in" קבוע. לחיצה עליו → מצב בחירה קצר:
1. המפה מקבלת overlay קליל
2. רק שולחנות **פנויים** ניתנים ללחיצה (אחרים מעומעמים)
3. לוחצים על שולחן פנוי → מודל קטן: "מספר סועדים + שם (אופציונלי)"
4. יוצרים reservation עם `is_walk_in=true`, `time=now()`, `table_id=<selected>`

זה שומר על הכלל "לחיצה רגילה על שולחן לא עושה כלום" אבל נותן דרך אינטואיטיבית ל-walk-in. אם זה לא מוצא חן, אלטרנטיבה: הכפתור פותח מודל עם dropdown של שולחנות פנויים (בלי אינטראקציה עם המפה).

## 3. Data Model

### טבלה חדשה: `restaurant_tables`

```sql
create table public.restaurant_tables (
  id uuid primary key default gen_random_uuid(),
  table_number int not null unique,          -- 1, 2, 101, 102...
  label text,                                -- אופציונלי: "בר קאונטר" וכו'
  shape text not null check (shape in ('square','rectangle','bar_stool')),
  width int not null default 80,             -- px במערכת הקואורדינטות של המפה
  height int not null default 80,
  pos_x int not null,                        -- px (top-left)
  pos_y int not null,
  capacity_min int not null default 1 check (capacity_min >= 1),
  capacity_max int not null default 2 check (capacity_max >= capacity_min),
  area text not null default 'table' check (area in ('bar','table')),
  -- area נשמר גם אם המפה מאוחדת: זה מתיישר עם reservations.area הקיים,
  -- ומאפשר בעתיד לחלק לוגיקה אם תרצה (לא משתמשים בזה היום).
  active boolean not null default true,      -- soft-delete כדי לא לאבד היסטוריה של הזמנות ישנות
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create index tables_active_idx on restaurant_tables (active) where active = true;
alter table restaurant_tables enable row level security; -- deny-all, עובר דרך service role כמו reservations
```

### שינויים ב-`reservations`:

```sql
alter table public.reservations add column table_id uuid references public.restaurant_tables(id) on delete set null;
alter table public.reservations add column is_walk_in boolean not null default false;

create index reservations_table_id_idx on public.reservations (table_id);
create index reservations_walk_in_idx on public.reservations (is_walk_in) where is_walk_in = true;
```

- `table_id` הוא nullable: הזמנות ישנות וגם הזמנות שהאלגוריתם לא מצא עבורן שולחן יישארו null.
- `is_walk_in=true` → walk-in שנוצר ע"י המארחת (בלי email/phone חובה; נרפה בחלק מהאילוצים בשכבת ה-DB).

## 4. אלגוריתם השיוך האוטומטי

מופעל כש-`POST /api/reservations` (יצירת הזמנה באתר).

```
קלט: date, time, guests, area (bar/table)
פלט: table_id | null

1. שולפים את כל restaurant_tables (active=true)
2. מסננים: capacity_min <= guests <= capacity_max
3. לכל שולחן שעבר — בודקים conflict:
   - תחום הזמן של ההזמנה החדשה: [time, time + RESERVATION_DURATION_MINUTES]
   - מסננים החוצה שולחנות שכבר יש להם הזמנה חופפת ב-OCCUPYING_STATUSES
4. אם נשאר יותר מאחד — ממיינים לפי best-fit:
   - עדיפות ראשונה: capacity_max הקטן ביותר (לא לבזבז מלבן של 8 על זוג)
   - עדיפות שנייה: table_number (דטרמיניסטי)
5. מחזירים את הראשון, או null אם אין
```

**שלב זה נכנס לתוך `lib/capacity.ts`** (פונקציה חדשה: `findTableForReservation`) — זה טבעי כי הלוגיקה של capacity ו-availability כבר שם.

**אם המשיכה מחזירה null:** ההזמנה נוצרת בכל זאת (כדי לא לשבור את flow הבקינג הקיים) עם `table_id=null`, והאדמין מקבל אינדיקציה בולטת ברשימת ההזמנות + במפה.

**הערה על הבר:** כיום `capacity.ts` שומר `MAX_BAR_PARTY=3` — אם הזמנה לבר היא קבוצה של 3, האלגוריתם צריך למצוא 3 שרפרפים *סמוכים* או שולחן בר בקיבולת 3. לא קריטי לגרסה 1, אפשר לעשות "כל עוד יש 3 שרפרפים פנויים באותה שעה — שיוך לראשון שבהם, עדיפות לסמוכים". נפתח נקודה זו ב-v2.

## 5. שיטת קביעת הסטטוס במפה

הסטטוס **לא נשמר** בטבלת השולחנות — הוא **נגזר בזמן ריצה** מתוך reservations.

לכל שולחן, בהינתן "עכשיו" (`now` בשעון ישראל):
- **תפוס (אדום):** יש reservation עם `table_id=<זה>`, `status='arrived'`, ו-`now` בתוך [start, end]
- **מוזמן (כתום/צהוב):** יש reservation עם `table_id=<זה>`, `status IN ('pending','confirmed')`, ו-`now` בתוך [start - 15min, end]
- **פנוי (ירוק/אפור נייטרלי):** אחרת

ה-endpoint `GET /api/admin/map/state?at=<ISO>` יחזיר JSON:
```json
{
  "generated_at": "...",
  "tables": [
    { "id": "...", "table_number": 1, "status": "reserved", "reservation_id": "..." },
    ...
  ]
}
```

**רענון:** polling כל 60 שניות בדף המפה (פשוט), עם אופציה לעבור ל-Supabase Realtime ב-v2.

## 8. Endpoints חדשים

| Method | Path | תיאור |
|---|---|---|
| `GET` | `/api/admin/map/tables` | רשימת כל השולחנות (לרנדור המפה) |
| `POST` | `/api/admin/map/tables` | יצירת שולחן חדש |
| `PATCH` | `/api/admin/map/tables/:id` | עדכון (מיקום/גודל/קיבולת/מספר) |
| `DELETE` | `/api/admin/map/tables/:id` | soft-delete (active=false) |
| `GET` | `/api/admin/map/state?at=ISO` | סטטוסים חיים של כל השולחנות בשעה נתונה |
| `POST` | `/api/admin/walk-ins` | יצירת walk-in (ראה סעיף 2) |

כל ה-endpoints עוברים דרך `admin-auth.ts` הקיים.

## 10. שלבי ביצוע מומלצים

### Phase 1 — תשתית ועורך (~3-4 ימי עבודה)
- DB migration: `restaurant_tables`, עמודות חדשות ב-`reservations`
- Endpoints CRUD לשולחנות
- עמוד `/admin/map` — view בלבד, כל השולחנות בצבע "פנוי"
- מצב עריכה: פאלטה, drag, פאנל עריכה, שמירה
- כפתור "מפה" בסיידבר

**בסיום Phase 1:** אפשר לבנות את המפה, לשמור אותה, ולראות אותה — בלי שום אינטגרציה להזמנות.

### Phase 2 — סטטוסים חיים (~1-2 ימים)
- `GET /api/admin/map/state`
- צבעים במפה לפי סטטוס
- Polling של דקה
- מקרא (legend) בתחתית

### Phase 3 — שיוך אוטומטי (~2 ימים)
- `findTableForReservation` ב-`capacity.ts`
- אינטגרציה ל-POST של הזמנה באתר
- אינדיקציה ב-`/admin/reservations` למי שלא שויך

### Phase 4 — Walk-ins (~1-2 ימים)
- כפתור + מצב בחירה במפה
- `POST /api/admin/walk-ins`
- הצגה ברשימת ההזמנות

### Phase 5 — Override ידני (~0.5 יום)
- dropdown של שולחנות בדף ההזמנה הבודדת
- אזהרה אם השולחן החדש בקונפליקט

**סה"כ אומדן:** 8-10 ימי עבודה לגרסה מלאה. Phase 1 לבד כבר נותן ערך (אפשר לבנות ולראות את המפה).

## 11. סיכונים וסייגים

- **ביצועים:** אם מגיעים ל-50+ שולחנות + הרבה הזמנות, ה-`findTableForReservation` צריך להיות יעיל. כרגע זה לולאה פשוטה — לא צפויה בעיה ל-scale הזה.
- **גבולות ה-canvas:** צריך לקבוע גודל מפה (למשל 1200×800 px). מה שקורה אם מזיזים שולחן מחוץ לתחום? — קלמפ-ים למלבן.
- **קונפליקט בעריכה:** אם שני אנשים עורכים בו-זמנית (לא צפוי כי רק מנהל), ה-last write wins. לא ננהל optimistic locking ב-v1.
- **מעבר מהלוגיקה הישנה:** `BAR_CAPACITY`/`TABLE_CAPACITY` ב-env — לאחר Phase 1 אפשר להעביר את הקיבולת להיות **סכום של restaurant_tables פעילים**. זה שינוי נפרד; לא חובה בגרסה 1.
- **טיימזון:** כל החישובי "עכשיו" חייבים לעבור דרך `shift-day.ts` כדי לכבד את חתך 04:00.
